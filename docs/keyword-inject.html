Keyword detection → when inject fires

Your reading of Fablize is right. Here’s how Vertex compares, and exactly how our keyword detection works and when text is injected.

Your understanding: Fablize’s important part is UserPromptSubmit — short mode nudge every user prompt, plus router one-liners when keywords match (investigation / grounding), pointing at pack files.

We do have keyword detection and we do inject mode + investigation/grounding/review from it — but the timing and shape differ: we classify on chat.message, then re-attach those directives on every following LLM call via system.transform until the next user message reclassifies. We do not use short “follow pack file” pointers; we inline full procedure text. Static contract is separate (agent / slash), not the UPS-style short block.

1. Fablize vs Vertex — the “UPS / router” piece

Fablize (Claude UserPromptSubmit)

  • When: every user prompt submit
  • Mode nudge: a few lines (“task mode: deep…”)
  • Router: if prompt contains debug/html/… → one line + path to pack file
  • Pack body: stays on disk; model is told to follow the file
  • Not re-run on every intermediate LLM step the same way

Vertex (OpenCode)

  • When classify: each chat.message (user message), if harness on
  • Mode nudge: short-ish verification-required / advisory
  • Router-like: same message → debugging / render / review flags
  • Pack body: full investigation/grounding/review text inlined
  • When inject: every system.transform (each model call) until next user message

2. Vertex timeline (one user message)

1. You send a message → chat.message · Is harness on? (agent = elicify-vertex-agent OR /elicify-vertex was used) · If no → stop. No classify, no inject. · If yes → run keyword detectors on the message text classifyStopMode(text) → quick | normal | deep + risks classifyTask(text) → baseline | build | debugging | render isReviewTask(text) → true | false · Store results on the session · Maybe show once: [vertex] harness on · stopMode=… 2. Model is about to answer → system.transform (may run several times per turn) · Read stored classification · Inject dynamic directives into system prompt (invisible in chat UI): if stopMode normal/deep → verification-advisory | verification-required if task debugging → full investigation procedure if task render → full grounding procedure if review → review-recall procedure if ledger has evidence → ledger one-liner if queue has items → tool-failure / stop-block / … · Does NOT re-inject full static [vertex:contract] (that’s agent/slash) 3. Tools run → tool.execute.after · Update scorecard; maybe queue failure text (delivered on next system.transform) 4. Model goes idle → session.idle · Maybe visible stop / promise continuation (not keyword-based)

4. Worked examples

User: "hi"
→ stopMode=quick, task=baseline, review=false
system.transform: often empty (or almost) static contract: only if agent md or earlier /elicify-vertex slash activate cue: once if first activation
User: "deep implement auth for production"
→ stopMode=deep (deep + auth + production risks), task=build, review=false
system.transform injects: [vertex:verification-required] Deep task mode. Risk flags: production, secret-or-auth. …
User: "debug why the test is failing"
→ stopMode=normal (fix/debug keywords), task=debugging
system.transform injects: [vertex:verification-advisory] … [vertex:investigation] full 6-step procedure (inlined)
User: "build an HTML dashboard chart"
→ stopMode=normal/deep (build), task=render
system.transform injects: [vertex:verification-…] … [vertex:grounding] full run/observe/fix loop (inlined)
User: "review this PR for security flaws"
→ review=true (+ maybe deep if security)
system.transform injects: [vertex:review-recall] collect then filter (+ verification-required if deep)

5. What is not keyword-driven

Inject / actionTrigger
Static [vertex:contract] Agent prompt always, or /elicify-vertex slash expansion — not keyword router
tool-failure / repeat-failure Tool exit ≠ 0 (and repeat signature)
stop-block session.idle + deep + files changed + no verify
promise-no-act session.idle + last assistant text matches promise patterns
ledger Scorecard non-empty this turn
Bottom line: Yes — we do keyword-detect on the user message (like Fablize UPS + router). Difference: Fablize injects a short tip at prompt submit and points at pack files; we store the classification and inject full procedures on every system transform until the next user message. The “most important” Fablize-style short UPS block is closest to our verification-required/advisory + investigation/grounding — not to the old every-turn full contract (removed in 0.9.3).
