Side-by-side map of OpenCode hooks, the closest Claude Code hook, what Fablize does, and what elicify-vertex does — in plain English — plus the exact style of text each side injects (if any).

Vertex flow (simple)

You send a message → turn harness on/off, reset scorecard, maybe show “harness on” A tool finishes → note file changes / tests; maybe queue a failure reminder Model is about to answer → put rules in the system prompt (you usually don’t see them) Model goes idle (“done”) → if proof is missing, block and send a visible nudge File edited / compact → extra change signal / don’t lose queued reminders

vertex:contract (exact body)[vertex:contract] Verification reminder: before reporting a task as done, - observe the actual output of the change (run the test, render the artifact, hit the endpoint); - ground any "done" claim in a tool result from this turn, not in intent; - if a step failed and you cannot fix it, surface that explicitly. What counts as verification: an allowlisted test, lint, typecheck, build, check, validate, verify, or HTTP probe… Silent successful tools such as tsc count. A Write/Edit success message is authoring, not verifying. A passing test is not evidence until you have confirmed the test can fail. …control it yourself… Communicate in a calm, factual tone…
Deep mode add-on (exact)[vertex:verification-required] Deep task mode. Define the exit proof before completion and verify changed behavior before the final response. A deep turn that changes no files needs no verification note; changed non-documentation files require observed successful verification.

Optional mode prompts Vertex may add every turn (system)

When classified Inject id What it tells the model (summary)
Always (active) vertex:contract Verify before “done”; calm tone; Write ≠ verify
stopMode = normal vertex:verification-advisory If files change, run one check or say why not (does not hard-block)
stopMode = deep vertex:verification-required Must define exit proof; non-docs changes need observed verify
task = debugging vertex:investigation Full 6-step investigate discipline (reproduce → 3+ hypotheses → …)
task = render/UI vertex:grounding Full run → observe → fix → re-run loop
task = review vertex:review-recall Collect all findings first, filter in a second pass
Evidence this turn vertex:ledger One line: files changed / verified counts / risks
Biggest difference: Fablize keeps the always-on text short in CLAUDE.md and injects short mode lines on UserPromptSubmit. Vertex puts a full contract (and often full procedures) into the system prompt on every model turn while active. Both use tool results + stop/idle for the “you’re not done” gates.
